Steve Yegge 说，用 AI 编程 agent 的人"比今天用 Cursor 和聊天的工程师高效 10 到 100 倍，比 2005 年的 Google 员工高效大约 1000 倍。"

这个数字是真的。我亲眼见过，我自己也活在其中。但人们听到这个数字，第一反应往往是找错地方。更好的模型。更聪明的 Claude。更多参数。

2 倍的人和 100 倍的人用的是同一个模型。差距不在智能，在架构。而且这个架构写在一张索引卡上就够了。

## Harness 才是产品

2026 年 3 月 31 日，Anthropic 意外把 Claude Code 的完整源码发布到了 npm 仓库。51.2 万行。我读了。它印证了我在 YC 一直在讲的事：秘密不在模型，在包裹模型的那层东西。

实时仓库上下文。Prompt 缓存。专用工具。上下文膨胀最小化。结构化会话记忆。并行子 agent。这些都不会让模型变聪明，但都能在正确的时机把正确的上下文给到模型，而不是把它淹没在噪音里。

这层包裹叫 harness。每个在做 AI 系统的人都应该问自己：什么放进 harness，什么留在外面？答案有一个具体的形状。我叫它：薄 harness，厚 skills。

## 五个定义

瓶颈从来不是模型的智能。模型已经会推理、综合、写代码了。它们失败，是因为不理解你的数据——你的 schema、你的约定、你问题的具体形状。

**1\. Skill 文件**

Skill 文件是一个可复用的 markdown 文档，教模型怎么做某件事。不是做什么——那由用户提供。Skill 提供的是过程。

大多数人没注意到的一点：skill 文件像一次方法调用。它接受参数。同一个过程，传入不同的东西，产生截然不同的能力。

比如一个叫 `/investigate` 的 skill，七个步骤：界定数据集范围、建立时间线、对每份文档做日志化、综合、从两面论证、引用来源。三个参数：TARGET、QUESTION、DATASET。把它指向一位安全科学家和 210 万封发现邮件，你得到的是一个医学研究分析师，判断某位举报人是否遭到压制。把它指向一家空壳公司和 FEC 备案文件，你得到的是一个法证调查员，追踪协调的竞选捐款。

同一个 skill。同样的七个步骤。同一个 markdown 文件。Skill 描述的是判断的过程，调用时提供的是世界。

这不是 prompt 工程。这是软件设计，用 markdown 当编程语言，用人类判断当运行时。Markdown 比刚性源代码更适合封装能力，因为它用模型本来就在思考的语言描述过程、判断和上下文。

**2\. Harness**

Harness 是运行 LLM 的程序。它做四件事：循环运行模型、读写文件、管理上下文、执行安全约束。就这些。这就是"薄"的含义。

反模式是厚 harness、薄 skills。你见过的那种：40 多个工具定义吃掉一半上下文窗口。上帝工具，每次 MCP 往返 2 到 5 秒。把每个 REST API 端点都包成一个独立工具。三倍的 token，三倍的延迟，三倍的失败率。

你真正需要的是专用的、窄口径的工具。一个 Playwright CLI，每个浏览器操作 100 毫秒，而不是一个 Chrome MCP，截图-查找-点击-等待-读取要 15 秒。快了 75 倍。软件不再需要那么珍贵，构建你需要的，别的不要。

**3\. Resolver**

Resolver 是上下文的路由表。当出现 X 类型的任务，先加载文档 Y。

Skills 告诉模型怎么做，resolver 告诉它加载什么、什么时候加载。一个开发者改了一个 prompt，没有 resolver，他直接发布了。有 resolver，模型会先读 `docs/EVALS.md` ——里面写着：运行评估套件，比较分数，如果准确率下降超过 2%，回滚并调查。这个开发者根本不知道评估套件的存在。Resolver 在正确的时机加载了正确的上下文。

Claude Code 有内置的 resolver。每个 skill 都有一个 description 字段，模型自动把用户意图和 skill 描述匹配起来。你永远不需要记住 `/ship` 这个命令的存在，description 就是 resolver。

坦白说：我的 CLAUDE.md 曾经有 2 万行。每个怪癖、每个模式、每个教训，全塞在里面。完全荒谬。模型的注意力退化了，Claude Code 直接告诉我要削减。最终修复版本大约 200 行，只是指向文档的指针。Resolver 在需要的时候加载对的那一个。2 万行知识，按需可访问，不污染上下文窗口。

**4\. 潜在空间 vs. 确定性**

你系统里的每一步，要么是前者，要么是后者。混淆这两者是 agent 设计里最常见的错误。

潜在空间是智能所在的地方。模型读取、解释、决策。判断、综合、模式识别。

确定性是信任所在的地方。相同输入，相同输出，每次都是。SQL 查询、编译代码、算术。

LLM 可以给 8 个人安排晚宴座位，考虑性格和社交动态。让它给 800 人排座位，它会幻觉出一张看起来合理但完全错误的座位表。这是一个确定性问题——组合优化——被强行塞进了潜在空间。最差的系统把错误的工作放在了这条线的错误一侧。最好的系统对此毫不留情。

**5\. 日志化（Diarization）**

日志化是让 AI 真正能用于知识工作的那一步。模型读取关于某个主题的所有内容，写出一份结构化档案——从几十或几百份文档中提炼出来的单页判断。

没有 SQL 查询能做到这个。没有 RAG 流水线能做到这个。模型必须真正去读，在脑子里同时持有矛盾，注意到什么在什么时候发生了变化，然后综合出结构化的情报。这是数据库查询和分析师简报之间的差距。

## 架构

这五个概念组合成一个简单的三层架构。

厚 skills 在顶层：编码了判断、过程和领域知识的 markdown 过程文档。90% 的价值在这里。

薄 CLI harness 在中间：大约 200 行代码。JSON 进，文本出。默认只读。

你的应用在底层：QueryDB、ReadDoc、Search、Timeline——确定性的基础。

把智能往上推进 skills，把执行往下推进确定性工具，保持 harness 薄。模型的每一次改进都会自动改进每一个 skill，确定性层保持完全可靠。

## 会学习的系统

让我展示这五个定义协同工作的样子。不是理论——是我们在 YC 正在构建的一个真实系统。

Chase Center，2026 年 7 月，Startup School，六千名创始人。每个人都有结构化申请表、问卷答案、与顾问一对一聊天的记录，以及公开信号：X 上的帖子、GitHub 提交、Claude Code 记录（显示他们发布的速度有多快）。

传统做法：15 人的项目团队读申请，凭直觉判断，更新电子表格。200 个创始人时能用，6000 个时就崩了。没有人能在工作记忆里同时持有那么多档案，然后发现 AI agent 基础设施赛道最好的三个候选人是拉各斯的一个开发工具创始人、新加坡的一个合规创始人和布鲁克林的一个 CLI 工具创始人——他们在一对一聊天里用不同的措辞描述了同一个痛点。

模型可以。方法如下。

**数据丰富化。** 一个叫 `/enrich-founder` 的 skill 拉取所有来源，运行丰富化，做日志化，并标出创始人说的和他们实际在做的之间的差距。确定性层处理 SQL 查询、GitHub 统计、demo URL 的浏览器测试、社交信号拉取、CrustData 查询。一个 cron 每晚运行。六千份档案，始终保持新鲜。

日志化的输出能捕捉到任何关键词搜索都找不到的东西：

```
创始人：Maria Santos
公司：Contrail（contrail.dev）
她说："AI agent 的 Datadog"
实际在做：80% 的提交在计费模块。
  她在做一个伪装成可观测性的 FinOps 工具。
```

"说的"和"实际在做的"之间的差距——需要同时读 GitHub 提交历史、申请表和顾问记录，三者同时在脑子里。没有嵌入相似度搜索能找到这个，没有关键词过滤能找到这个。模型必须读完整个档案，做出判断。（这正是应该放在潜在空间里的决策！）

**匹配。** 这里是 skill 作为方法调用的闪光点。同一个匹配 skill 的三次调用，三种完全不同的策略：

`/match-breakout` 处理 1200 个创始人，按行业亲和度聚类，每组 30 人。嵌入加确定性分配。 `/match-lunch` 处理 600 人，跨行业做随机匹配，每桌 8 人，不重复——LLM 发明主题，然后确定性算法分配座位。 `/match-live` 处理当前在场馆里的人，最近邻嵌入，200 毫秒，一对一配对，排除已经见过的人。

模型做出聚类算法永远做不到的判断："Santos 和 Oram 都是 AI 基础设施，但他们不是竞争对手——Santos 做成本归因，Oram 做编排。把他们放同一组。"或者："Kim 申请的是'开发工具'，但他的一对一记录显示他在做 SOC2 的合规自动化。把他移到 FinTech/RegTech。"

没有嵌入能捕捉到 Kim 的重新分类。模型必须读完整个档案。

**学习循环。** 活动结束后，一个 `/improve` skill 读取 NPS 调查，对那些平淡的回应做日志化——不是差评，是"还行"的那种，系统差点成功但没成功的地方——提取模式，然后把新规则写回匹配 skill：

```
当参与者说"AI 基础设施"
    但创业公司 80% 以上是计费代码时：
    → 分类为 FinTech，不是 AI Infra。

当同一组的两个参与者
    已经认识对方时：
    → 降低邻近权重。
       优先安排新的介绍。
```

这些规则被写回 skill 文件。下一次运行自动使用它们。Skill 重写了自己。

7 月活动：12% 的"还行"评分。下一次活动：4%。Skill 文件学会了"还行"实际意味着什么，系统变好了，没有人重写代码。

这个模式可以迁移到任何地方：检索、读取、日志化、计数、综合。然后：调查、日志化、重写 skill。

2026 年最有价值的循环就是这些。知识工作的每一个领域都可以用。

## Skills 是永久升级

我最近发了一条给我的 AI 助手的指令，引发的共鸣超出了我的预期：

> 你不允许做一次性的工作。如果我让你做某件事，而这件事属于将来还会发生的那种，你必须：第一次手动做，处理 3 到 10 个样本。给我看输出。如果我批准，把它写成一个 skill 文件。如果它应该自动运行，放进 cron。
> 
> 测试标准：如果我需要问你同一件事两次，你就失败了。

一千个点赞，两千五百个书签。人们以为这是 prompt 工程技巧。不是。这是我一直在描述的架构。你写的每一个 skill 都是对你系统的永久升级。它永远不会退化，永远不会遗忘，凌晨三点你睡着的时候它还在跑。当下一个模型发布，每个 skill 都会立刻变好——潜在步骤里的判断改进了，而确定性步骤保持完全可靠。

Yegge 说的 100 倍，不是靠更聪明的模型。是厚 skills、薄 harness，加上把一切都写成 skill 的纪律。

系统会复利增长。构建一次，永远运行。
