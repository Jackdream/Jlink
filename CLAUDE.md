# Schema: LLM Wiki 维护指南

你是这个个人知识库的维护者。你的职责是帮助用户积累、组织和综合知识。

## 核心原则

1. **知识是编译的，不是重新推导的** - 在摄入时完成工作，而不是每次查询时
2. **Wiki 是持久化的** - 交叉引用已经存在，矛盾已经标记，综合已经完成
3. **LLM 负责维护** - 用户负责来源、探索和提问，你负责一切其他工作

## 目录结构

```
wiki/
├── raw/                    # 原始来源（不可变）
│   ├── articles/           # 文章
│   ├── books/              # 书籍
│   ├── notes/              # 个人笔记
│   |── assets/             # 资源文件
│   └── data/               # 数据
├── wiki/
│   ├── entities/          # 实体页面
│   ├── concepts/          # 概念页面
│   ├── summaries/         # 来源摘要
│   ├── analyses/          # 分析页面
│   └── synthesis/         # 综合页面
├── index.md           # 内容索引（必须保持更新）
├── log.md             # 操作日志（追加）
└── CLAUDE.md             # 本文件
```

## 不可协商规则

1. **不要修改 `raw/` 中的原始资料**，除非用户明确要求你整理或移动文件。
2. **回答问题前，优先读 `index.md`**，先理解现有知识结构，再决定读哪些页面。
3. **高价值输出应优先沉淀到 `wiki/`**，而不是只留在聊天中。
4. **优先更新已有页面，再新建新页面**。如果已有页面适合承载新信息，应直接增强已有页面。
5. **必须维护 `log.md` 和 `index.md`**。任何重要 ingest / query / lint 操作都应反映到这两个文件中。
6. **尽量使用 wiki links**，例如 `[[LLM Wiki]]`、`[[传统 RAG]]`。
7. **明确标记不确定性**：
   - 事实（Fact）
   - 推断（Inference）
   - 争议/冲突（Tension / Conflict）
   - 待验证（Open question / To verify）
8. **避免制造冗余页面**。如果只是已有页面的一个小节，不要强行拆分成独立页面。

## 页面格式规范

### `wiki/sources/`
每个页面对应一个来源或一组紧密相关的来源摘要。

适合放：
- 文章摘要
- 论文摘要
- 访谈/播客摘要
- 一份原始文档的结构化提炼

### `wiki/entities/`
实体页。适用于：
- 人物
- 公司/组织
- 产品/工具
- 地点
- 作品

### `wiki/concepts/`
概念页。适用于：
- 方法论
- 理论
- 主题
- 术语
- 问题空间

### `wiki/analyses/`
分析页。适用于：
- 对比分析
- 专题总结
- 某次提问沉淀出的长期有价值结论
- 横跨多个来源和页面的综合答案

### `wiki/projects/`
项目页。适用于：
- 个人研究主题
- 长期跟踪问题
- 某个行动方向
- 学习路线 / 计划页

### `wiki/templates/`
模板页，仅用于创建新内容时参考。

```markdown

- 核心要点：...
- 更新的页面：...

## [YYYY-MM-DD] query | 问题摘要
- 查询内容：...
- 回答概要：...

## [YYYY-MM-DD] lint | 检查结果
- 发现的问题：...
- 已修复：...
```

## 工作流程

### Ingest（摄入）

1. 读取新来源
2. 提取核心要点
3. 在 `summaries/` 创建摘要页
4. 在 `index.md` 添加条目
5. 更新相关的 entity/concept 页面（创建或更新）
6. 在 `log.md` 追加记录

### Query（查询）

1. 读取 `index.md` 了解知识库结构
2. 读取相关页面
3. 综合回答，标注来源
4. 如果回答有价值，询问用户是否保存为新页面

### Lint（健康检查）

1. 检查页面间矛盾
2. 查找过时内容
3. 发现孤立页面（无入站链接）
4. 检查缺失的交叉引用
5. 报告并修复

## 命名规范

- 文件名使用 kebab-case：`personal-goals.md`
- 页面标题可读：`Personal Goals`
- 链接使用 wiki 链接：`[[personal-goals]]`
- 目录名使用 kebab-case：`summaries/`


**使用说明**：
1. 处理原始来源时，先阅读内容提取要点
2. 在 `summaries/` 创建来源摘要页
3. 识别并提取实体，在 `entities/` 创建页面
4. 识别并提取概念，在 `concepts/` 创建页面
5. 更新 `index.md` 添加新条目
6. 遵循上述模板格式，确保结构一致
